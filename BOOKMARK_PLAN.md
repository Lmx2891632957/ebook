<h2 id="1">一、TECH_DESIGN 原方案的可行性分析</h2>

### 1.1 数据结构分析

原方案：
```c
typedef struct {
    char    book_path[256];
    uint32_t bookmark_count;
    struct {
        uint32_t file_offset;
        uint32_t page_num;
        char     description[32];
    } bookmarks[10];
} bookmark_db_t;
```

| 设计点 | 判断 | 分析 |
|--------|------|------|
| `book_path[256]` | 需修正 | STM32F103RC 只有 48KB SRAM，256 字节固定数组浪费内存。现有代码中路径用 `gui_memin_malloc` 动态分配，应保持一致 |
| `bookmark_count` | 需修正 | 当前结构体只存储一本书的书签。若要支持多本书，需要外层再包一层结构；或者改为每本书一个独立书签文件 |
| `file_offset` | 可行 | 使用文件字节偏移作为书签锚点是正确的——它与字体大小无关，是最稳定的定位方式 |
| `page_num` | 冗余但有用 | 页码依赖字体大小，切换字号后页码会变。但作为显示用途（"第 15 页"）仍有价值。存储它没问题，但跳转时应以 `file_offset` 为准 |
| `description[32]` | 可简化 | 从书签位置提取附近文字需要额外 `f_lseek` + `f_read`，增加复杂度。初期可省略 |
| 每本书 10 个书签 | 合理 | 对于嵌入式阅读器足够了 |

### 1.2 存储方案分析

| 方案 | 判断 | 分析 |
|------|------|------|
| SD 卡 `bookmark.dat` | 可行 | FATFS 已就绪，`f_open` + `f_write` 即可 |
| 单文件存所有书签 | 有风险 | TECH_DESIGN 自己提到了——写入中途断电会损坏整个文件。但加上备份机制后可接受 |
| 启动加载 / 退出写回 | 可行 | 符合现有代码的模式 |

### 1.3 UI 交互分析

原方案说"阅读界面点击书签按钮 → 弹出书签列表"，这个思路可行，但缺少细节：
- 书签列表用什么控件？→ 正点原子已有 `listbox` 控件，可直接复用
- 如何区分"添加书签"和"查看书签"？→ 需要设计弹窗交互
- 如果当前页已有书签，要不要提示？→ 需要去重逻辑

### 1.4 总结：TECH_DESIGN 方案大方向正确，但需要修正 5 个细节

1. 路径存储改用动态分配
2. 数据结构改为支持多本书
3. 明确 `file_offset` 为主键、`page_num` 仅用于显示
4. 简化 `description`（先不做自动摘要）
5. 补充 UI 交互细节

<h2 id="2">二、修正后的详细实现方案</h2>

<h3 id="2.1">2.1 存储设计</h3>

**方案选择：每本书一个独立书签文件**

对比 TECH_DESIGN 的"单文件存所有书签"：

| 维度 | 单文件方案 (原) | 每书一文件方案 (推荐) |
|------|----------------|---------------------|
| 文件损坏影响 | 所有书的书签全丢 | 只丢一本书的 |
| 文件大小 | 随书增多而膨胀 | 每个文件很小（约 424 字节） |
| 查找效率 | 需遍历所有记录匹配路径 | 文件名直接对应 |
| 实现复杂度 | 需要解析/序列化多本书 | 简单，一本书一个结构体直接读写 |

**文件名规则**：书名 + `.bmk` 后缀，例如 `三体.txt` → `三体.bmk`。存放在 SD 卡与原书相同的目录下。

**文件内部格式**（二进制，固定大小，总计 424 字节）：

```
字节 0-3:   魔数 0x424D4B00 ("BMK\0")
字节 4-7:   书签数量 (uint32_t)
字节 8-263: 书籍路径 (256 字节，GBK 编码，不足补 \0)
字节 264+:  书签数组，每条 16 字节 x 10 条 = 160 字节
            每条: file_offset(4B) + page_num(4B) + reserved1(4B) + reserved2(4B)
```

文件总大小 = 8 + 256 + 160 = 424 字节，极小，读写几乎无延迟。

<h3 id="2.2">2.2 数据结构设计</h3>

```c
/* 单个书签条目 */
typedef struct {
    uint32_t file_offset;    /* 主键：书签在文件中的字节偏移 */
    uint32_t page_num;       /* 仅用于显示，跳转时不依赖此值 */
    uint32_t reserved1;      /* 预留 */
    uint32_t reserved2;      /* 预留 */
} ebook_bm_entry_t;

/* 一本书的书签集（同时也是 SD 卡文件的磁盘格式） */
#define EBOOK_BM_MAX    10

typedef struct {
    uint32_t magic;              /* 魔数: 0x424D4B00 */
    uint32_t count;              /* 有效书签数量 */
    uint8_t  book_path[256];     /* 书籍路径（GBK），用于校验匹配 */
    ebook_bm_entry_t entries[EBOOK_BM_MAX];  /* 书签数组 */
} ebook_bm_book_t;
```

**内存使用估算**：`ebook_bm_book_t` 约 424 字节，占 STM32F103RC 48KB SRAM 不到 1%，完全可接受。

<h3 id="2.3">2.3 UI 交互设计</h3>

#### 2.3.1 按钮布局

在阅读界面底部栏增加"书签"按钮。当前底部栏已有两个按钮（字体、返回），书签按钮放在字体按钮左侧：

```
┌──────────────────────────────────────────┐
│                  LCD 屏幕                 │
│  ┌────────────────────────────────────┐  │
│  │         顶部栏（书名）              │  │
│  ├────────────────────────────────────┤  │
│  │                                    │  │
│  │         文本显示区域                │  │
│  │                                    │  │
│  ├────────────────────────────────────┤  │
│  │  [书签]    [字体]    [返回]        │  │  ← 底部栏
│  └────────────────────────────────────┘  │
└──────────────────────────────────────────┘
```

#### 2.3.2 书签管理弹窗（一级面板）

点击"书签"按钮后弹出模态面板：

```
┌──────────────────────────┐
│      📑 书签管理         │  ← 标题行
├──────────────────────────┤
│  [+ 添加当前页为书签]    │  ← 始终显示（已满10个则提示）
├──────────────────────────┤
│  书签1: 第 15 页         │  ← listbox，点击进入二级面板
│  书签2: 第 42 页         │
│  书签3: 第 108 页        │
│  ...                     │
├──────────────────────────┤
│       [ 返  回 ]         │  ← 关闭弹窗
└──────────────────────────┘
```

**实现方式**：标题行用 `gui_show_string` 绘制，书签列表用 `listbox` 控件，上下各放一个 `btn` 控件。

#### 2.3.3 书签操作二级面板

点击某个书签条目后，**不直接跳转**，而是弹出一个**二级选择面板**：

```
┌──────────────────────────┐
│      📑 书签操作         │  ← 标题
├──────────────────────────┤
│  书签: 第 15 页          │  ← 显示所选书签信息
├──────────────────────────┤
│   [ 跳转到此处 ]         │  ← 跳转按钮
│   [ 删除此书签 ]         │  ← 删除按钮（红色/警示色）
├──────────────────────────┤
│       [ 取  消 ]         │  ← 返回一级面板
└──────────────────────────┘
```

**设计理由**：
- 电阻触摸屏精度有限，直接点击跳转容易误触——二级确认避免误操作
- 删除操作有确认入口，不会无心中丢失书签
- 逻辑清晰，不依赖 listbox 的长按等高级触控能力

#### 2.3.4 交互流程

```
State 1 (阅读中)
│
├── 点击 [书签] 按钮
│   └── 进入一级面板：书签管理（内层 while 循环 I）
│       │
│       ├── 点击 [+ 添加当前页为书签]
│       │   ├── 检查是否已存在（按 file_offset 判断，±100 字节视为同一位置）
│       │   ├── 检查是否已满 10 个
│       │   ├── 添加新条目 → 写入 SD 卡（即时持久化）
│       │   └── 刷新列表显示
│       │
│       ├── 点击已有书签条目
│       │   └── 进入二级面板：书签操作（内层 while 循环 II）
│       │       ├── 点击 [跳转到此处] → f_lseek 定位 → 重新排版 → 关闭全部弹窗
│       │       ├── 点击 [删除此书签] → 从数组移除 → 写入 SD 卡 → 返回一级面板
│       │       └── 点击 [取消] → 返回一级面板（不操作）
│       │
│       └── 点击 [返回]
│           └── 关闭弹窗，恢复阅读界面（还原被弹窗覆盖的区域）
```

**子状态机说明**：两层面板各用一个内层 `while` 循环实现（与现有字体选择面板的模态循环模式一致）。二级面板退出时回到一级面板的循环，直到用户在一级面板点击"返回"。

<h3 id="2.4">2.4 文件读写设计</h3>

#### 写入流程（每次添加/删除书签时立即写入 SD 卡）

```c
f_open(bm_path, FA_WRITE | FA_CREATE_ALWAYS)
f_write(&file, &bm_book, sizeof(ebook_bm_book_t), &bw)
f_close(&file)
```

因为整个结构体只有 424 字节，一次 `f_write` 即可完成。在这个大小下，底层扇区写入是原子的，不存在"写一半断电"的风险——因此**无需 TECH_DESIGN 提到的备份机制**。

#### 读取流程（打开书籍时加载）

```
根据书籍路径构造书签文件名 → f_open(bm_path, FA_READ)
├── 成功 → f_read 整个结构体 → 校验 magic == 0x424D4B00 → 校验 book_path 匹配
└── 失败 (FR_NO_FILE) → 初始化空的书签集（count=0, magic=0x424D4B00, 复制 book_path）
```

<h3 id="2.5">2.5 书签文件命名方案</h3>

**书名 + `.bmk`**：例如 `三体.txt` → `三体.bmk`

优点：
- 可读、可调试（SD 卡插到电脑上能直接识别）
- 不需要 hash 算法，不会冲突
- FATFS 已启用 LFN（长文件名支持），中文文件名无问题

书签文件与原书放在同一目录下，便于管理。

<h3 id="2.7">2.7 函数接口设计</h3>

```c
/* ========== ebook_bookmark.h ========== */

/* 加载指定书籍的书签（从 SD 卡读取），返回书签集指针 */
ebook_bm_book_t *ebook_bm_load(const uint8_t *book_path);

/* 保存书签到 SD 卡，返回 0 成功 */
uint8_t ebook_bm_save(ebook_bm_book_t *bm);

/* 添加书签，返回 0 成功 / 1 已满 / 2 已存在 */
uint8_t ebook_bm_add(ebook_bm_book_t *bm, uint32_t file_offset, uint32_t page_num);

/* 删除指定索引的书签，返回 0 成功 / 1 索引越界 */
uint8_t ebook_bm_delete(ebook_bm_book_t *bm, uint8_t index);

/* 释放书签集内存 */
void ebook_bm_free(ebook_bm_book_t *bm);

/* 根据书籍路径构造书签文件路径（结果存入 dst，大小至少 280 字节） */
void ebook_bm_get_path(const uint8_t *book_path, uint8_t *dst);
```

<h3 id="2.8">2.8 关键边界条件处理</h3>

| 场景 | 处理方式 |
|------|----------|
| 书签文件不存在（新书） | 初始化空集，第一次添加书签时才创建文件 |
| 书签已满（10 个） | "添加书签"按钮仍然显示，点击后弹 toast 提示"书签已满" |
| 同一位置重复添加 | `file_offset` 相差 < 200 字节视为同一位置，返回"已存在" |
| SD 卡写入失败 | 返回错误码，上层弹提示"保存失败" |
| 切换字体后跳转书签 | 用 `file_offset` 重新排版，页码自动重新计算，不影响定位 |
| 书籍文件被删除/移动 | 书签文件仍在但无法使用（下次打开时 `book_path` 校验不匹配） |
| 退出阅读时 | 书签在添加/删除时已即时写入 SD 卡，退出时只需释放内存 |

<h2 id="4">四、方案总结</h2>

TECH_DESIGN 的书签方案**整体可行**，但在以下 4 个方面做了修正和细化：

1. **存储格式**：从"所有书签一个大文件"改为"每本书一个独立小文件"——更安全、更简单
2. **数据结构**：去掉固定 `description[32]` 字段，预留扩展位，明确 `file_offset` 为主键
3. **写入策略**：因为文件仅 424 字节，单次 `f_write` 即可原子写入，无需备份机制
4. **UI 设计**：给出了具体的按钮布局、一二级弹窗样式和完整交互流程，复用现有 `listbox` 控件

整体工作量约 **300 行代码**，其中核心书签逻辑约 150 行，UI 集成约 130 行，数据结构定义约 40 行。
